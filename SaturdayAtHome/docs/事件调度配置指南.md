## 1. 架构总览

一个 block 的完整事件序列由三个层次组成：

```
game_config.yaml
    │  声明所有时间参数（基准时间、jitter 范围、间隔范围）
    ▼
block_scheduler.generate_block_schedule(block_num, condition, seed)
    │  用确定性 RNG 生成完整 ScheduledEvent 列表
    │  → 应用 resolve_conflicts() 检测并修复冲突
    │  → 将审计事件写入 block_events 表（scheduled_t 列）
    ▼
BlockTimeline.run()
    │  按 ScheduledEvent.t 顺序顺序执行
    │  → 每次 dispatch 后更新 block_events.actual_t
    ▼
send_sse(session_id, sse_event_name, payload)
    │  经 SSE_EVENT_MAP 映射为前端期望的事件名
    ▼
前端 useSSE.js
```

> **关键约束**：`game_config.yaml` 是单一事实来源。所有时间参数都从这里读取，**不要在 Python 代码里硬编码任何秒数**。

## 2. `game_config.yaml` — `timeline` 节点完整说明

```yaml
timeline:
  block_duration_s: 510       # Block 总时长（秒）。block_end 事件在这个时刻触发。

  steak_spawn:                # 牛排生成器
    start_s: 3                # 第一块牛排的生成时刻
    end_s: 490                # 最后一次生成截止时刻（含）
    interval_min_s: 8         # 两次生成的最小间隔（秒）
    interval_max_s: 15        # 两次生成的最大间隔（秒）

  plant_water:                # 植物浇水提示
    start_s: 45               # 第一次提示的基准时刻
    start_jitter_s: 15        # 第一次提示在 [start_s, start_s + jitter] 内随机
    interval_min_s: 40        # 后续提示的最小间隔
    interval_max_s: 60        # 后续提示的最大间隔

  events:
    # ── 浮动实验事件（有 jitter 的）────────────────────────────────────
    fake_trigger_s: 35
    fake_trigger_jitter_s: 8         # 实际触发时刻 ∈ [27, 43]s

    robot_neutral_1_s: 75
    robot_neutral_1_jitter_s: 12     # 实际触发时刻 ∈ [63, 87]s

    force_yellow_1_s: 95
    force_yellow_1_jitter_s: 8       # 实际触发时刻 ∈ [87, 103]s
    force_yellow_1_hob: 0            # 强制哪口锅进入 READY_SIDE1

    robot_neutral_2_s: 270
    robot_neutral_2_jitter_s: 12     # 实际触发时刻 ∈ [258, 282]s

    force_yellow_2_s: 275
    force_yellow_2_jitter_s: 8       # 实际触发时刻 ∈ [267, 283]s
    force_yellow_2_hob: 1

    # ── 固定 PM 事件（无 jitter）────────────────────────────────────────
    reminder_a_s: 120                # Reminder A 播放时刻
    trigger_a_appear_s: 210          # PM 触发器 A 出现（执行窗口打开）
    trigger_a_close_s: 240           # PM 执行窗口 A 关闭（30s 后）

    reminder_b_s: 300                # Reminder B 播放时刻
    trigger_b_appear_s: 390          # PM 触发器 B 出现
    trigger_b_close_s: 420           # PM 执行窗口 B 关闭

  neutral_comments:            # robot_neutral 事件携带的文本池（顺序对应 idx 0, 1）
    - "Smells great in the kitchen!"
    - "How's everything coming along?"

  messages:                    # 消息气泡（固定时刻，不参与 jitter）
    - time_s: 55
      from: "Sarah"
      ...
```

### 2.1 参数类型速查

| 参数后缀 | 含义 | 代码读取方式 |
|---|---|---|
| `_s` | 基准时刻（秒，相对 block 开始） | `float(ev.get("..._s", default))` |
| `_jitter_s` | 以基准为中心的单侧半宽（秒） | `_jitter(rng, base, half)` → 均匀分布 |
| `_min_s` / `_max_s` | 随机间隔的上下界（秒） | `rng.randint(min, max)` |
| `_ms` | 毫秒值（用于前端 / cooking duration） | 直接透传到 SSE payload |
| `_hob` | 锅编号（0 = 左，1 = 中，2 = 右） | 直接透传到 SSE payload |

## 3. 固定事件 vs 浮动事件

### 3.1 固定事件（`is_fixed=True`）

触发时刻由配置直接决定，**不受 seed 影响**，每次 block 相同：

| 事件 | 说明 |
|---|---|
| `block_start` | t=0，block 开始 |
| `reminder_fire`（A 和 B） | `reminder_a_s` / `reminder_b_s`，机器人说提醒语 |
| `trigger_appear`（A 和 B） | `trigger_a_appear_s` / `trigger_b_appear_s`，打开执行窗口 |
| `window_close`（A 和 B） | `trigger_a_close_s` / `trigger_b_close_s`，关闭执行窗口 |
| `block_end` | `block_duration_s`，block 结束 |
| `message_bubble` ×5 | `messages[i].time_s`，消息气泡 |

> 固定事件是 PM 实验设计的锚点，时刻变动会影响 retention interval 的一致性，修改前必须更新 PRD。

### 3.2 浮动事件（`is_fixed=False`）

触发时刻 = 基准 ± jitter，由 seeded RNG 决定，同 seed 可完全重现：

| 事件 | 基准参数 | jitter 参数 | 有效区间（默认） |
|---|---|---|---|
| `fake_trigger_fire` | `fake_trigger_s` | `fake_trigger_jitter_s` | [27, 43]s |
| `robot_neutral` ×1 | `robot_neutral_1_s` | `robot_neutral_1_jitter_s` | [63, 87]s |
| `force_yellow_steak` ×1 | `force_yellow_1_s` | `force_yellow_1_jitter_s` | [87, 103]s |
| `robot_neutral` ×2 | `robot_neutral_2_s` | `robot_neutral_2_jitter_s` | [258, 282]s |
| `force_yellow_steak` ×2 | `force_yellow_2_s` | `force_yellow_2_jitter_s` | [267, 283]s |
| `steak_spawn` ×N | loop，`start_s` + 随机间隔 | `interval_min/max_s` | 全程 |
| `plant_needs_water` ×N | `start_s` + `start_jitter_s` | `interval_min/max_s` | 全程 |

## 4. 如何调整事件时序

### 4.1 移动一个固定事件

直接修改 `game_config.yaml` 中对应的 `_s` 值：

```yaml
# 把 Reminder A 从 120s 移到 100s（更早播放）
reminder_a_s: 100

# 把执行窗口 A 的打开时刻跟着移（保持 reminder→trigger 间隔不变）
trigger_a_appear_s: 190   # 原来 210，相差 90s → 移后同样相差 90s
trigger_a_close_s: 220    # 保持 30s 窗口
```

**注意联动关系**：

```
force_yellow_1_s  <  reminder_a_s  <  trigger_a_appear_s  <  trigger_a_close_s
     ↑                  ↑                    ↑
  力黄必须先于           提醒                窗口打开
  提醒前完成
```

移动任何一个，检查相邻事件是否仍然满足上述偏序关系。

### 4.2 调整浮动事件的基准位置

```yaml
# 把 fake trigger 从 [27,43]s 移到 [40,55]s
fake_trigger_s: 47
fake_trigger_jitter_s: 8   # 区间宽度不变，中心移动
```

修改 jitter 半宽可以控制"随机幅度"：

```yaml
# 几乎固定（几乎不随机）
fake_trigger_jitter_s: 1   # ±1s，接近确定性

# 高度随机
fake_trigger_jitter_s: 20  # ±20s，每次 block 差异明显
```

> **实验设计提示**：jitter 的意义是避免被试建立"第 N 秒一定有事件"的期待，同时保证 reminder → trigger 的 retention interval 恒定。一般推荐 jitter < 0.25 × (reminder_s - fake_trigger_s)，避免两者发生时刻重叠。

### 4.3 调整执行窗口长度

执行窗口 = `trigger_a_close_s - trigger_a_appear_s`。当前默认 30s，在 `game_config.yaml` 的 `experiment` 节也有独立记录：

```yaml
experiment:
  execution_window_ms: 30000  # 前端和后端都读这个字段
```

但 block_scheduler.py 以 `trigger_*_close_s - trigger_*_appear_s` 的实际差值为准。**两处必须保持同步**：

```yaml
trigger_a_appear_s: 210
trigger_a_close_s: 250    # 改为 40s 窗口

experiment:
  execution_window_ms: 40000  # 同步修改
```

### 4.4 调整牛排生成密度

```yaml
steak_spawn:
  interval_min_s: 5   # 更快（认知负荷提高）
  interval_max_s: 10
```

```yaml
steak_spawn:
  interval_min_s: 15  # 更慢（认知负荷降低）
  interval_max_s: 25
```

生成总数估算：`block_duration / avg(interval)` ≈ `510 / 11.5 ≈ 44` 次（默认参数）。

### 4.5 改变事件顺序

事件按 `ScheduledEvent.t` 升序排列，**顺序由时刻决定，不是代码中的声明顺序**。

若想让 `neutral_comment_1` 在 `fake_trigger` 之前触发：

```yaml
robot_neutral_1_s: 25    # 基准移到 25s（比 fake_trigger 的 [27,43]s 更早）
robot_neutral_1_jitter_s: 4   # 区间缩到 [21,29]s，确保不超过 27s
fake_trigger_s: 35
fake_trigger_jitter_s: 5
```

> 修改后必须运行冲突检测逻辑（见第 5 节）验证新排列不违反保护规则。

## 5. 冲突检测与保护窗口

### 5.1 三条冲突规则

`block_scheduler.resolve_conflicts()` 在每次生成 schedule 时自动运行，检测以下三条规则：

#### 规则 1：力黄不能在 PM 执行窗口内

`FORCE_STEAK_READY` 的触发时刻不能落在 `[trigger_appear - 5s, window_close]` 区间内。

**原因**：执行窗口期间要求被试专注于 PM 任务识别，此时牛排突然变黄会分散注意力，也会影响被试的 ongoing task 认知负荷对照。

```
违规示例：
  trigger_a_appear_s = 210, trigger_a_close_s = 240
  force_yellow_1_s 的 jitter 样本 = 207   ← 落在 [205, 240] 内 → 违规

修复：
  force_yellow_1_s 被推移到 210 - 5 - 1 = 204s
```

触发这条规则的条件：`w_open - 5 <= e.t <= w_close`

修复方式：`e.t = w_open - 6.0`（保护边界外 1s）

#### 规则 2：假触发不能贴近 Reminder

`FAKE_TRIGGER` 的触发时刻距 reminder 时刻的绝对距离不能小于 10s。

**原因**：假触发和 reminder 同时出现会形成"注意力竞争"，影响 reminder 被听到/处理的概率，破坏 AF 条件的操控效度。

```
违规示例：
  reminder_a_s = 120
  fake_trigger 的 jitter 样本 = 114   ← |114 - 120| = 6 < 10 → 违规

修复：
  fake_trigger 被推移到 120 - 10 - 2 = 108s
```

触发条件：`abs(e.t - rt) < 10`

修复方式：`e.t = rt - 12.0`

#### 规则 3：两次 Neutral Comment 间距 ≥ 30s

**原因**：机器人连续说两句话（即使内容随机）会让被试产生"机器人在聊天"的感觉，削弱单次评论的自然性。

```
违规示例：
  neutral_1 jitter 样本 = 78
  neutral_2 jitter 样本 = 102   ← 102 - 78 = 24 < 30 → 违规

修复：
  neutral_2.t → 78 + 30 = 108s
```

### 5.2 自定义冲突规则

在 `block_scheduler.py` 的 `resolve_conflicts()` 中添加新规则：

```python
# 示例：message_bubble 不能在执行窗口内出现
# （message_bubble 是固定时刻，只记录警告，不强制移动）
for e in events:
    if e.event_type == EventType.MESSAGE_BUBBLE:
        for w_open, w_close in trigger_windows:
            if w_open <= e.t <= w_close:
                logger.warning(
                    f"message_bubble at {e.t}s overlaps trigger window "
                    f"[{w_open}, {w_close}] — adjust messages[].time_s in config"
                )
```

对于固定事件，代码**不应自动移动**（因为这会破坏研究设计的对称性），只记录警告，由研究者手动修正配置。

## 6. 确定性随机（Seed 系统）

### 6.1 Seed 计算

```python
# block_scheduler.py
def derive_seed(session_id: str, block_num: int) -> int:
    import hashlib
    raw = hashlib.md5(f"{session_id}:{block_num}".encode()).hexdigest()[:8]
    return int(raw, 16)
```

- 同一 `(session_id, block_num)` 永远产生同一 seed → 同一 schedule
- 不同 session 的 seed 不同（保证被试间 schedule 差异）
- 不同 block 的 seed 不同（保证块间 schedule 差异）

### 6.2 Seed 的持久化

`block_events` 表的每一行都存有生成该 schedule 使用的 seed：

```sql
SELECT seed, scheduled_t, actual_t, event_type, payload
FROM block_events
WHERE session_id = 'abc123' AND block_num = 1
ORDER BY scheduled_t;
```

### 6.3 重放（Replay）

用相同的 seed 重新生成 schedule，可得到完全相同的事件序列：

```python
# 重放 session abc123 的 block 1
seed_from_db = 0xA3F1C2B4  # 从 block_events 表读取
events = generate_block_schedule(1, "HighAF_HighCB", seed_from_db)
# events 与原始 schedule 完全一致
```

### 6.4 指定 Seed 测试

在 session.py 的 `block_stream()` 中，`BlockTimeline.__init__` 调用 `derive_seed()`。若需要固定 seed 做 QA：

```python
# 临时：在 BlockTimeline.__init__ 里覆盖 seed
self.seed = 42  # 固定 seed，用于重复测试
self.schedule = generate_block_schedule(block_num, condition, self.seed)
```

## 7. 时间精度分析

### 7.1 精度来源与误差

调度引擎使用顺序循环 + `asyncio.sleep()`：

```python
wait = event.t - (time.monotonic() - block_start)
if wait > 0:
    await asyncio.sleep(wait)
```

| 误差来源 | 典型幅度 | 说明 |
|---|---|---|
| `asyncio.sleep()` 本身 | ±1–5 ms | Linux 系统级 timer 精度 |
| Event loop 调度延迟 | ±5–30 ms | 其他协程（DB 写入、SSE 广播）插入时 |
| 满载时的竞争延迟 | 最大约 80 ms | 多会话并发时，event loop 繁忙 |
| DB `actual_t` 写入滞后 | ±10 ms | `_update_actual_t()` 是额外 I/O |

**结论**：在单会话条件下，绝大多数事件的实际触发误差 < 50 ms，对 8.5 分钟 block 而言可忽略不计。

### 7.2 时间基准

```python
block_start = time.monotonic()  # 单调时钟，不受系统时间调整影响
actual_t = time.monotonic() - block_start  # 相对时间，与 scheduled_t 对比
```

- `time.monotonic()` 不受 NTP 同步、夏令时等影响，适合测量间隔
- `scheduled_t` 和 `actual_t` 都是"距 block 开始的秒数"，直接相减得漂移量

### 7.3 精度监控

通过 `block_events` 表查询每个 session 的时间漂移：

```sql
-- 查看所有审计事件的调度误差（秒）
SELECT
    session_id,
    block_num,
    event_type,
    scheduled_t,
    actual_t,
    ROUND(actual_t - scheduled_t, 3) AS drift_s
FROM block_events
WHERE actual_t IS NOT NULL
  AND session_id = 'abc123'
ORDER BY scheduled_t;
```

期望输出：
```
session_id | block_num | event_type          | scheduled_t | actual_t | drift_s
abc123     | 1         | block_start         | 0.0         | 0.002    | 0.002
abc123     | 1         | fake_trigger        | 29.3        | 29.318   | 0.018
abc123     | 1         | reminder            | 120.0       | 120.034  | 0.034
abc123     | 1         | trigger_window_open | 210.0       | 210.041  | 0.041
```

漂移超过 500 ms 视为异常（通常意味着服务器过载或网络堵塞）。

### 7.4 Busy-Window 门控对精度的影响

`TRIGGER_WINDOW_OPEN` 事件在厨房处于高负荷状态时（有牛排正在 BURNING）会被延迟，最多等待 10s：

```python
if _is_busy(session_id):
    await _wait_for_not_busy(session_id, timeout=10.0)
```

这是**有意设计的误差**，属于实验操控的一部分：
- `scheduled_t` 记录的是计划时刻
- `actual_t` 记录的是实际触发时刻
- 两者之差即为门控引入的延迟，事后可从 `block_events` 表提取

若研究设计**不允许**执行窗口的时刻浮动，注释掉 `timeline.py` 中的 busy-window 检查即可：

```python
# 移除此段以禁用 busy-window 门控
# if event.event_type == EventType.TRIGGER_WINDOW_OPEN:
#     if _is_busy(self.session_id):
#         await _wait_for_not_busy(self.session_id, timeout=10.0)
```

### 7.5 前一版本（并发 gather）vs 现版本（顺序循环）的精度对比

| 指标 | 旧版：`asyncio.gather()` | 新版：顺序循环 |
|---|---|---|
| 理论精度 | 各任务独立 sleep，精度相同 | 同左 |
| 实际漂移积累 | 每个 sleep 误差独立，不叠加 | 每个 sleep 误差独立，不叠加 |
| `actual_t` 记录 | ❌ 无，无法事后比对 | ✅ 每个审计事件都记录 |
| 服务器内存 | 同时 ~50 个 Task 对象 | 同时 1 个协程，内存更低 |
| Busy-window 门控 | ❌ 不支持（gather 并发，无法条件等待） | ✅ 原生支持 |

## 8. 添加新事件类型

### 8.1 第一步：在 `event_schedule.py` 中添加 EventType

```python
class EventType(str, Enum):
    ...
    MY_NEW_EVENT = "my_new_event"   # 新增
```

### 8.2 第二步：添加 SSE 事件名映射

```python
SSE_EVENT_MAP: dict[EventType, str] = {
    ...
    EventType.MY_NEW_EVENT: "my_new_event",  # 前端 useSSE 处理的事件名

```

若不需要推送到前端（内部状态标记），不加入 `SSE_EVENT_MAP`，`timeline.py` 会跳过 dispatch。

### 8.3 第三步（可选）：加入审计日志

```python
AUDITED_EVENT_TYPES: frozenset[EventType] = frozenset({

timeline:

    my_new_event_jitter_s: 10   # 如果是浮动事件
```

### 8.5 第五步：在 `block_scheduler.generate_block_schedule()` 中生成事件

```python
# 固定事件
events.append(ScheduledEvent(

# 或浮动事件
events.append(ScheduledEvent(
    event_type=EventType.MY_NEW_EVENT,
    t=_jitter(rng, float(ev.get("my_new_event_s", 180)),
                   float(ev.get("my_new_event_jitter_s", 10))),
    payload={"key": "value"},
    is_fixed=False,
))
```

elif event == "my_new_event":
    # 例如：更新内存状态、打开新窗口等

## 9. 跨 Block 状态重置

每次新 block 的 SSE stream 建立时（`/session/{id}/block/{n}/stream`，`auto_start=True`），系统自动执行：

```python
reset_session_hobs(session_id)     # 3 口锅全部清空 → EMPTY
reset_session_windows(session_id)  # 上一 block 的执行窗口全部清除
```

这发生在 `BlockTimeline.__init__()` 之前。时序：

```
block_stream() 被调用
    ↓
reset_session_hobs()      ← 锅清空
reset_session_windows()   ← 窗口清除
    ↓
BlockTimeline.__init__()
    derive_seed()
    generate_block_schedule()   ← 新 schedule 生成
    _persist_schedule()         ← 写入 block_events（scheduled_t）
    ↓
asyncio.create_task(tl.run())   ← 顺序执行开始
```

### 9.1 如需保留跨 Block 状态

注释掉 `session.py` 中对应的 reset 调用：

```python
# 保留牛排状态跨 block（如研究设计需要持续性）
# reset_session_hobs(session_id)

# 保留执行窗口跨 block（罕见，通常不需要）
# reset_session_windows(session_id)
```

## 11. Latin Square 与 Reminder 文本配置

### 11.1 Condition 命名约定

实验采用 2×2 被试内设计，四个条件的代码如下：

| 代码 | AF（Aftereffects） | CB（Cue Busyness） |
|---|---|---|
| `LowAF_LowCB` | Low | Low |
| `HighAF_LowCB` | High | Low |
| `LowAF_HighCB` | Low | High |
| `HighAF_HighCB` | High | High |

### 11.2 Latin Square 矩阵

`game_config.yaml` 的 `experiment.latin_square` 节定义四个组的 block-condition 对应关系：

```yaml
experiment:
  latin_square:
    A: ["LowAF_LowCB", "HighAF_LowCB", "LowAF_HighCB", "HighAF_HighCB"]
    B: ["HighAF_LowCB", "LowAF_HighCB", "HighAF_HighCB", "LowAF_LowCB"]
    C: ["LowAF_HighCB", "HighAF_HighCB", "LowAF_LowCB", "HighAF_LowCB"]
    D: ["HighAF_HighCB", "LowAF_LowCB", "HighAF_LowCB", "LowAF_HighCB"]
```

每行的索引即对应 block 1–4。例如 B 组被试：block 1 = HighAF_LowCB，block 2 = LowAF_HighCB，以此类推。

### 11.3 被试分组算法

`assign_group()` 在创建 session 前读取数据库中已有的 session 数量，用于取模：

```python
count = SELECT COUNT(*) FROM sessions
group = ["A","B","C","D"][count % 4]
```

- 基于 DB 计数，**服务器重启后分组顺序保持连续**（不依赖内存计数器）
- 删除测试 session 会影响后续计数，正式数据收集前应清库

### 11.4 Reminder 文本格式（per-slot）

每个 condition 下，Slot A 和 Slot B 的 reminder 文本独立配置：

```yaml
experiment:
  reminder_texts:
    LowAF_LowCB:
      A: "By the way, remember — after dinner today, take your medicine."
      B: "Don't forget — after dinner, take your medicine."
    HighAF_LowCB:
      A: "By the way, remember — after dinner today, take your Doxycycline..."
      B: "Don't forget — after dinner, take your Doxycycline..."
    LowAF_HighCB:
      A: "I can see you're keeping an eye on the stove. By the way — ..."
      B: "You're busy in the kitchen — don't forget, after dinner, ..."
    HighAF_HighCB:
      A: "I can see you're keeping an eye on the stove. By the way — ..."
      B: "You're busy in the kitchen — don't forget, after dinner, ..."
```

**AF 维度**（高/低提醒具体性）由 A 槽和 B 槽**分别**控制文本内容，两条提醒可以有不同的措辞：
- Slot A reminder → 在 `reminder_a_s`（120s）播放，对应 PM task A
- Slot B reminder → 在 `reminder_b_s`（300s）播放，对应 PM task B

> 旧格式（单字符串 `LowAF_LowCB: "..."`)仍被 `block_scheduler.py` 向后兼容，但不推荐。

### 11.5 管理员强制启动 Block

`POST /api/admin/force-block/{session_id}/{block_num}` 允许实验员不通过参与者端 SSE 连接，直接从 Dashboard 启动某一 block 的完整时间线：

- 取消该 block 已有的 timeline（如存在）
- 重置 hob 和执行窗口状态
- 按该被试在 DB 中记录的 `condition_order` 选取正确的 condition
- 创建并启动 `BlockTimeline`

**Dashboard 操作**：Controls 面板底部"⚡ Force-Start Block N"按钮，点击后需二次确认。

典型使用场景：
- 被试端页面崩溃重载后，实验员需要重启当前 block
- Pilot 阶段手动控制 block 进程不依赖参与者操作

## 12. 快速参考：配置检查清单

修改 timeline 参数后，依次核对：

```
□ force_yellow_1_s + jitter_max < reminder_a_s           （力黄先于提醒）
□ reminder_a_s + 10 < trigger_a_appear_s                 （提醒到窗口有充足间隔）
□ trigger_a_appear_s + execution_window_ms/1000 = trigger_a_close_s
□ trigger_a_close_s < reminder_b_s - 10                  （窗口 A 关闭后再播 reminder B）
□ force_yellow_2_s + jitter_max < reminder_b_s           （同上）
□ trigger_b_close_s < block_duration_s                   （窗口 B 在 block 内完成）
□ steak_spawn.end_s < block_duration_s - 20              （最后一批牛排有时间处理）
□ fake_trigger_s ± jitter 不与任何 reminder_*_s ± 10s 重叠
□ robot_neutral_2 - robot_neutral_1 > 30s（含 jitter 最坏情况）
□ messages[i].time_s 不落在 [trigger_*_appear_s, trigger_*_close_s] 内
```

所有检查通过后，用 `generate_block_schedule()` 打印完整时间轴做视觉审查。
